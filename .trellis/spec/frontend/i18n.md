# i18n (Paraglide)

> Executable contract for internationalization: Paraglide message catalog, locale fallback, menu-title dynamic resolution, and BA error translation.

---

## 1. Scope / Trigger

Triggers when work touches any of:

- Any user-facing `.tsx` text (JSX literals, alt text, aria-label, placeholder, toast message, form label)
- `messages/zh.json` / `messages/en.json` (message catalog)
- `project.inlang/settings.json` (locale config / baseLocale / plugin paths)
- `src/components/layout/AppSidebar.tsx` menu label resolution
- `src/lib/auth-errors.ts` (BA error code → Chinese translator)
- `src/emails/*.tsx` (email copy also goes through Paraglide)
- Adding a new locale or deprecating one

Generated artifacts `src/paraglide/*` are **read-only** — regenerate via the Vite plugin, never hand-edit.

## 2. Signatures

### Paraglide message exports (`#/paraglide/messages`)

```ts
import * as m from "#/paraglide/messages";

// Static access — preferred. TS verifies the key at compile time.
m.common_loading();                     // "加载中..."
m.email_subject_verify();               // "请验证你的邮箱"

// With params
m.email_invite_preview({ inviterName, organizationName });
```

Keys with dots (e.g. `menu.dashboard`) compile to string-named exports on the `m` namespace — **not** nested objects.

### Dynamic menu label resolver (`src/components/layout/AppSidebar.tsx`)

```ts
function resolveMenuLabel(title: string | undefined): string | undefined {
  if (!title) return undefined;
  if (title.startsWith("menu.") && title in m) {
    const fn = (m as unknown as Record<string, () => string>)[title];
    if (typeof fn === "function") return fn();
  }
  return title;   // fallback: operator-created menus keep literal titles
}
```

Only the `menu.` prefix triggers resolution. Titles without the prefix render verbatim, letting operators add menus via the admin UI without forcing them into the message catalog.

### BA error translator (`src/lib/auth-errors.ts`)

```ts
export interface AuthErrorLike {
  code?: string | null;
  message?: string | null;
}

export function translateAuthError(
  error: AuthErrorLike | null | undefined,
): string;

// Usage:
const { error } = await authClient.signIn.email({ email, password });
if (error) toast.error(translateAuthError(error));
```

- Looks up `error.code` (uppercased) in a static `CODE_TO_MESSAGE` map pointing to Paraglide fns.
- Falls back to `error.message` (raw BA English) if code is absent / unknown.
- Final fallback: `m.common_unknown_error()` — never returns empty string or undefined.

## 3. Contracts

### Project config (`project.inlang/settings.json`)

```jsonc
{
  "baseLocale": "zh",
  "locales": ["zh", "en"],
  "plugin.inlang.messageFormat": {
    "pathPattern": "./messages/{locale}.json"
  }
}
```

- `baseLocale: "zh"` — all Paraglide lookups fall back to `zh` when a locale is missing a key. `en.json` intentionally holds empty-value placeholders; the Chinese string renders until a translator fills it in.
- `locales` **must** list every `messages/*.json` file. Stray files (e.g. `messages/de.json` from the scaffold) must be deleted — Paraglide warns on compile otherwise.

### Rules for UI text

1. **No hard-coded CJK/EN in JSX/alt/aria/title/toast.** Every literal must be `m.xxx()`.
2. Add new keys to `messages/zh.json` **first**; Paraglide's type generation flags missing keys in `m.xxx()` call sites.
3. For parameterized copy, use `{name}` placeholders (Paraglide's `@inlang/plugin-message-format@4` flavour):
   ```json
   "email_invite_preview": "{inviterName} 邀请你加入 {organizationName}"
   ```
   The generated TS fn then requires `m.email_invite_preview({ inviterName, organizationName })`.
4. `messages/en.json` mirrors the zh key set with empty strings (or real translations). Never delete a key from one locale without deleting it from all.

### Menu `meta.title` convention

- Seeded menus store **i18n keys** starting with `menu.` (e.g. `menu.dashboard`, `menu.settings_organization`).
- Operator-created menus via the admin UI store plain strings. `resolveMenuLabel` is a no-op for those (no `menu.` prefix).
- Both coexist in one `Menu.meta.title` column — no schema split.

### Email templates

All copy in `src/emails/*.tsx` (subject, body, CTA, ignore-hint) goes through `m.email_*()`. Templates render during `sendEmail`, which runs server-side — Paraglide's static exports work identically on server and client.

## 4. Validation & Error Matrix

| Condition | Expected behaviour |
|---|---|
| Key present in `zh.json`, missing in `en.json` | Fallback to zh (per `baseLocale: "zh"`) at runtime; no TS error |
| Key missing in `zh.json` | TS error at `m.<key>()` call site (Paraglide fails compilation) |
| Menu `meta.title` does not start with `menu.` | `resolveMenuLabel` returns the literal → ok for operator menus |
| Menu `meta.title` starts with `menu.` but key missing from catalog | `in m` check fails → falls back to the raw key (e.g. `menu.ghost`) — visible bug, fix by adding the key |
| BA error `{ code: "INVALID_EMAIL_OR_PASSWORD" }` | `translateAuthError` returns `m.auth_error_invalid_email_or_password()` |
| BA error with unknown `code` and non-empty `message` | Returns the raw BA English message |
| BA error without `code` and without `message` | Returns `m.common_unknown_error()` |
| Stray `messages/de.json` file | Paraglide compile warning; build still produces `src/paraglide/messages.js` |
| Manual edit of `src/paraglide/*` | Lost on next `pnpm dev` / `pnpm build` (regenerated) |

## 5. Good / Base / Bad Cases

### Good

```tsx
// Component
<Button>{m.common_save()}</Button>

// Toast on BA error
toast.error(translateAuthError(error));

// Menu seed
meta: { title: "menu.settings_organization" }

// Email template
<Text>{m.email_verify_body()}</Text>
```

### Base

```tsx
// Operator-created menu via admin UI — literal title is fine
meta: { title: "业务日报" }  // resolveMenuLabel returns as-is
```

This is intentional: non-technical operators add menus at runtime; forcing them into the catalog would defeat the feature.

### Bad

```tsx
// Hard-coded string
<Button>保存</Button>                   // ❌ no m.xxx()
toast.error("邮箱或密码错误");          // ❌ use translateAuthError
<img alt="头像" />                       // ❌ aria/alt needs m.xxx()

// Hand-edit generated catalog
// src/paraglide/messages.js
export const common_save = () => "saved";  // ❌ lost on next build
```

## 7. Wrong vs Correct

### Wrong — hard-coded Chinese in JSX

```tsx
export function ConfirmDialog({ onConfirm }: Props) {
  return (
    <AlertDialog>
      <AlertDialogTitle>确认删除</AlertDialogTitle>
      <AlertDialogDescription>此操作不可撤销。</AlertDialogDescription>
      <AlertDialogCancel>取消</AlertDialogCancel>
      <AlertDialogAction onClick={onConfirm}>确认删除</AlertDialogAction>
    </AlertDialog>
  );
}
```

Problems:
1. Translator can't find the strings — `grep` in `.tsx` is not a localization workflow.
2. Future `en-US` rollout needs a full JSX sweep.
3. Copy changes require code edits + redeploy.

### Correct — Paraglide messages

```tsx
import * as m from "#/paraglide/messages";

export function ConfirmDialog({ onConfirm }: Props) {
  return (
    <AlertDialog>
      <AlertDialogTitle>{m.confirm_dialog_default_title()}</AlertDialogTitle>
      <AlertDialogDescription>{m.confirm_dialog_irreversible_hint()}</AlertDialogDescription>
      <AlertDialogCancel>{m.common_cancel()}</AlertDialogCancel>
      <AlertDialogAction onClick={onConfirm}>
        {m.confirm_dialog_default_confirm()}
      </AlertDialogAction>
    </AlertDialog>
  );
}
```

Rationale: one catalog (`messages/zh.json`), one compile-checked call site, one-file diff to add `en` translations.
