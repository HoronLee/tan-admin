# refactor(auth): modularize lib/auth + lib/email + add BA shadow codegen

## Goal

把当前 `src/lib/` 下扁平铺开的 auth / email 相关文件按职责收进**子目录模块**，并在此基础上引入 **Better Auth schema 影子自动生成机制**，让 ZenStack policy 引擎能正确感知 BA 表（同时不再手抄 zmodel 影子声明）。

最终交付物 = 一次性把项目从"方案 A 半吊子"推到"清晰的 lib 模块边界 + 方案 C 起点（最小契约影子 + 自动同步）"。

## Resolved Decisions

1. **`config.ts` 提取范围**（已定）：**单一 `config.ts` 真源，仅 `database` 不放进去**。
   - server.ts 和 codegen.ts 各自是 thin wrapper（`{ ...authConfig, database: pool / prismaAdapter }`）
   - 命名用 `config.ts` 而不是 `config.shared.ts`——"shared" 后缀冗余
   - 理由：BA 的 `generate` 不调 hooks，hooks 里的 `pool` 是模块顶层 import；让 codegen 加载到全部配置零副作用，换来"BA 配置全貌只看一个文件"

2. **prismaAdapter import 路径**：BA 当前主线版本里 `prismaAdapter` 应当来自 `better-auth/adapters/prisma`，需要在 `pnpm install` 完后实际跑一次 codegen 验证 import path——这个验证在实施时做，不阻塞 PRD

3. **drift 守卫**：先不加 CI 检查，靠开发者跑 `pnpm ba:shadow` 后 commit 影子文件，PR review 看 diff；如果未来发现频繁 drift，再补 GH Actions 跑 `pnpm ba:shadow && git diff --exit-code zenstack/_better-auth.zmodel`

## Design Principles (Non-negotiable)

- **单一职责**：每个文件只做一件事，文件名直接说明意图（`server.ts` / `client.ts` / `codegen.ts` / `config.ts`）
- **高内聚低耦合**：BA 相关全在 `lib/auth/`，邮件相关全在 `lib/email/`，跨目录引用最小化
- **不做 barrel**：调用方走深路径（`#/lib/auth/server`），看 import 一眼知道用了哪个子模块
- **薄壳厚芯**：runtime 入口（server.ts / codegen.ts）只做"组合 + 适配器注入"，业务逻辑全在 config.ts

1. 把现 server.ts 里**除 `database: pool` 外**的全部 BA option 整体抽到新文件 `config.ts`（export `authConfig`），server.ts 缩成 thin wrapper
2. 写 `codegen.ts`：import `authConfig` + 绑 prismaAdapter

4. 装 prismaAdapter 所需 deps（运行时若没引入 prisma，需 `pnpm add -D @prisma/client prisma`；codegen.ts 仅在 cli 流程加载，不影响生产 bundle）
5. 跑 `pnpm ba:shadow`，验证 `zenstack/_better-auth.zmodel` 生成
6. 编辑 `zenstack/schema.zmodel`：删 4 个手抄影子，加 `import "./_better-auth"`
7. 跑 `pnpm zen generate` + tsc + dev smoke
8. 更新 CLAUDE.md
9. commit：`feat(auth): add BA shadow schema codegen`

## Decision (ADR-lite)

**Context**: 当前 lib/ 用 `xxx-yyy.ts` 前缀凑命名空间，且 zmodel 半吊子声明 4 张 BA core 表（不含插件表），架构上既不内聚也不一致。后续业务复杂度升高（plan gating / banned user / 跨 org policy）需要 ZenStack 能感知 BA schema。

**Decision**: 一次性做 lib 模块化 + 方案 C 起点（最小影子契约 + codegen）。两件事虽然独立但前后置（codegen 输出要落到模块化后的 `src/lib/auth/codegen.ts`），合并一个 task / PR 但分两个 commit。

**Consequences**:
- 优点：lib 边界清晰；BA schema 自动同步；后续要做 FK 到 user / policy 引用 organization.plan 等只需扩展现有机制
- 风险：BA cli 输出的 Prisma schema 与我们假设有出入（rename 脚本可能要 tweak）；prismaAdapter 装包体积小幅增加（但仅 cli 流程加载）
- 回滚：两个 commit 各自独立，单步 revert 即可
